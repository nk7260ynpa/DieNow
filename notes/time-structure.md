## 背景與原始需求

使用者原始需求（繁體中文逐字保留）：

> 1. 透過程式重現漫畫：端腦 其中的「攜手之戒」關卡
> 2. 設計一個 Project Agent 這個 Agent 是該關卡的玩家
> 3. 設計一個系統 使玩家可以參與其中 且不會形成邏輯錯亂

漫畫資訊：

- 作品：《端腦》（Die Now）
- 作者：**壁水羽**（原 Coordinator 誤植為 Lingo，已修正）
- 關卡：「攜手之戒」（Ring of Joining Hands）

原作設定摘要（查證來源：維基百科／百度百科／萌娘百科）：

- 6 人參加，房間中央有 1–6 號按鈕
- 按對應按鈕會亮，**按錯即死**
- 面具（遮臉 + 變聲）+ 特殊眼鏡（**看不到自己的號碼牌**，可看見他人的）
- 6 燈齊亮後防護窗打開，**第一個觸碰戒指者**得到之
- 核心機制：「第一個人死後靈魂會自動寄宿在第二個人身上，以此類推；每一次復活，時間線都將重置」
- 關鍵悖論：「除了六號，前五號任意一號都不能拿走攜手之戒，否則時間線將會錯亂」
- 百度百科原文：「**六個人都為主角一人**」— 指主角夏馳透過靈魂寄宿輪迴體驗六個身份

## 時間結構：形式化 Ω（多時空階段的自我 × 單 Epoch）

### 核心概念

- **物理層**：場景有 6 個 body，全部活著、會動、會說話、會互相感知、可碰撞／讓路
- **意識層**：只有 **1 個 Project Agent**（1 個底層 LLM 模型 + 1 個 persona：「被困的玩家」）
- **時空切片**：Project Agent 的意識被時空切片為 6 份獨立的 `pov`（sub-agent contexts），每份佔一個 body
- 主觀感受上，每份 `pov` 都以為自己是「剛醒來的玩家」；他們**不知道**彼此是同一個人（戴面具 + 變聲）

```
               ┌─── 1 個底層 Project Agent ──────────┐
               │  Persona：「被困的玩家」              │
               │  Model：Claude Sonnet 4.7             │
               │  Anthropic SDK + prompt caching       │
               └────────────┬───────────────────────────┘
                            │
       ┌──────┬──────┬──────┴──────┬──────┬──────┐
       ▼      ▼      ▼             ▼      ▼      ▼
    pov_1  pov_2   pov_3         pov_4  pov_5  pov_6    ← 6 個獨立 context
       │      │      │             │      │      │
       ▼      ▼      ▼             ▼      ▼      ▼
    body_1 body_2 body_3         body_4 body_5 body_6   ← 6 個 body
    （彼此可見、可對話、可移動、可擋路）
```

### 初始狀態（關卡 t=0）：遞迴前世記憶

**關鍵修正**：關卡一開始，**只有 pov_1 是新生（無前世記憶）**，其他 5 個 pov 都帶層層遞迴的前世記憶。

| pov | 新生 | 前世記憶內容（直接） | 記憶深度（含遞迴） |
|-----|------|---------------------|-------------------|
| pov_1 | ✓ | 無 | 0 層 |
| pov_2 | — | 「死去的 body_1」的完整生命週期 | 1 層 |
| pov_3 | — | 「死去的 body_2」的完整生命週期 | 2 層（body_2 內含 body_1） |
| pov_4 | — | 「死去的 body_3」的完整生命週期 | 3 層 |
| pov_5 | — | 「死去的 body_4」的完整生命週期 | 4 層 |
| pov_6 | — | 「死去的 body_5」的完整生命週期 | 5 層（= 漫畫主角完成 5 次輪迴的狀態） |

```
pov_6 記憶 = 「死去的 body_5」
              = body_5 的完整人生（包括他在場時的觀察、行動、死亡）
              = 內含 body_5 的前世記憶「死去的 body_4」
              = 內含 body_4 的前世記憶「死去的 body_3」
              = 內含 body_3 的前世記憶「死去的 body_2」
              = 內含 body_2 的前世記憶「死去的 body_1」
              = 最底層 body_1 的完整人生（無前世）

 ∴ pov_6 在 t=0 就等同於漫畫結局狀態：
   「主角歷盡 5 次輪迴、終於到第 6 個身體的那一刻」
```

**意義**：關卡**開場狀態 = 漫畫結局狀態**。整個遊戲即是「這最後一次機會」。

### 關卡 = 單 Epoch（Q1 = a）

- **不存在 epoch 推進**；關卡只有一次機會
- **不存在時間線重置**；一場定勝負
- **通關條件**：body_6 在 6 燈齊亮後正確觸碰戒指
- **失敗條件**：
  - 非 body_6 的 pov 觸碰戒指（邏輯錯亂）
  - Timeout（達最大回合數仍未通關）
  - 通關必要條件無法滿足（例如 body_6 死亡、或必要按鈕的 body 死亡使燈數無法湊齊 6）

### 死亡不再接力記憶（Q2）

- 任何 pov 死亡（按錯 / 搶錯戒指 / 其他致死事件）→ 該 pov 死亡，body 變空殼
- **不觸發記憶接力**（因為大家本來就有該有的前世記憶，不需要傳遞）
- **不觸發時間線重置**（單 epoch 下無重置概念）
- 死亡的影響：
  - 該 body 的按鈕永遠不會亮 → 若湊不齊 6 燈 → 防護窗無法打開 → 本次關卡失敗
  - 該 body 不再說話、不再移動
  - 其他 pov 可觀察到該 body 處於「死亡」狀態

## 因果閉環（Q5）

### 核心要求

**時間邏輯必須通**。使用者舉例：

> pov_2 跟 pov_1 說了一段話，當 pov_1 成為了 pov_2 就要跟 pov_1 說話。

### 詮釋

在單 epoch + 時空切片模型下，「pov_1 成為 pov_2」是**抽象概念**：指在時間線上 pov_1 死後靈魂前進到 pov_2 階段。但實際上：

- pov_2 的前世記憶 = pov_1 的**完整生命週期**（從 t=0 醒來到死亡）
- 這條記錄必然會包含「某時刻被 pov_2 對自己說 speech_X」的事件
- 關卡執行時，pov_2 的確會對 pov_1 說 speech_X
- **時間一致性要求**：
  - pov_2 的前世記憶中「被 pov_2 說 speech_X 的時刻、措辭、內容」
  - 必須與關卡執行時 pov_2 對 pov_1 實際說出的 speech_X 完全一致

即：**pov_n 的前世記憶 = pov_{n-1} 完整人生的預先紀錄；該紀錄描述的所有互動事件，必須在關卡執行時如實發生**。

### 閉環保證的關鍵洞察

| 互動類型 | 是否受因果閉環約束 | 理由 |
|---|---|---|
| pov_1 ↔ pov_2 | **是** | 該互動會出現在 pov_3, 4, 5, 6 的前世記憶中 |
| pov_2 ↔ pov_3 | **是** | 該互動會出現在 pov_4, 5, 6 的前世記憶中 |
| … | … | … |
| pov_5 ↔ pov_6 | **否** ⭐ | pov_6 不是任何人的前世（沒有 pov_7） |
| pov_6 ↔ 任何人 | **否** ⭐ | 同上 |

⭐ **關鍵**：pov_6 的任何行為都**不受時間閉環約束**，因此可安全作為**真 Agent（自由意志）**，不會破壞系統邏輯一致性。

## 前世記憶設計方案（Q3）

### 架構：預生成閉環劇本 + pov_6 為真 Agent

#### 1. 劇本生成階段（關卡啟動前）

在關卡執行之前，系統先產生 `script_1, script_2, …, script_5` 共 5 份腳本：

```
Step 1: 生成 script_1
  輸入：pov_1 的 persona（無前世）
         世界環境描述（房間、6 body、按鈕、戒指、面具、眼鏡）
         提示：此 pov 從新生視角摸索
  LLM → 產出 script_1（pov_1 的完整生命週期：t=0 到死亡）

Step 2: 生成 script_2
  輸入：pov_2 的 persona
         script_1（作為 pov_2 的前世記憶）
         世界環境描述
  LLM → 產出 script_2（pov_2 的完整生命週期）
         ⚠ 需保證：script_2 中若有與 pov_1 的互動，必須與 script_1 一致

Step 3: 生成 script_3 ~ script_5（類似方式，遞迴疊加）

Step 6: pov_6 的前世記憶 = script_5（遞迴內含 script_1~4）
```

#### 2. 劇本內容格式

每份 `script_n` 包含：

- `persona`：該 pov 的人格描述（個性、說話風格、冒險傾向等；可配置/可隨機）
- `events`：時間序的事件清單，每筆事件：
  - `t`：時間戳（或回合編號）
  - `actor`：事件發起者（該 pov 自己 / 或他提及的他人）
  - `action_type`：`move` / `speak` / `press` / `touch_ring` / `observe` / `die` / ...
  - `payload`：動作內容（座標、訊息文字、按鈕編號等）
  - `targets`：互動對象（若有）
- `prior_life`：直接前世記憶（= `script_{n-1}` 的壓縮敘述或完整引用）
- `death_cause`：該 pov 的死亡原因（press_wrong / ring_paradox / timeout / ...）

#### 3. 時間一致性驗證

每產生一份 `script_n` 後，自動驗證：

- `script_n` 中涉及 pov_{n-1} 的所有事件，必須也存在於 `script_{n-1}` 中，且內容一致（時機、措辭、結果）
- 若驗證失敗 → 自動重新生成或微調（retry up to K 次）
- K 次仍失敗 → 由 Coordinator 於 `issues.md` 記錄，交人類介入

#### 4. 關卡執行階段

- **pov_1 ~ pov_5**：按**各自劇本嚴格行動**（不是 LLM 自由發揮，而是「按腳本演」）
  - 對話／細微反應可用 LLM 即時填充，但主幹（動作、時機、互動對象）必須符合劇本
  - 劇本為 immutable，執行時只讀
- **pov_6**：**真正的 Project Agent**（LLM + prompt caching，自由決策）
  - 收到完整 5 層前世記憶（`prior_life` = `script_5`）作為 system prompt
  - 每回合依觀察自行決策
  - 不受劇本約束（因為無 pov_7 依賴其行為）

#### 5. 「可預測角色未來互動」的實現

Project Agent（pov_6）可透過前世記憶 = `script_5`（遞迴內含 `script_1~4`）得知：

- pov_1 ~ pov_5 在關卡中的所有預定行為（何時走、何時說、何時按、何時死）
- 可預判他們之間的互動時機與結果
- 這是 pov_6 推理的極強素材；核心挑戰在於從中提煉出：
  1. 「我是 6 號」（透過排除法）
  2. 「只有 6 號能拿戒指」（透過觀察前世所有 pov 拿戒指都失敗）
  3. 正確的拿取時機

## 關卡核心張力（Q4）

**目標**：拿到戒指（且不造成邏輯錯亂）是唯一通關條件。

**張力來源**：pov_6 的「**認知旅程**」即是遊戲張力，儘管關卡是單 epoch：

1. 開場，Agent 以 pov_6 身份醒來，面對 5 層前世記憶的海量資訊
2. Agent 必須**解讀前世記憶**：
   - 自己經歷過的身份（1–5 號）
   - 其他 pov 的人格與行為模式
   - 關卡的潛在規則（按鈕、戒指、面具、眼鏡）
3. Agent 必須**推理自己的號碼**：
   - 觀察其他 5 個 body 的號碼牌（比方看到 1, 2, 3, 4, 5）
   - 排除法 → 「我是 6 號」
   - 但也要考慮：若某 body 已死（例如 body_3 按錯了），怎麼辨識？
4. Agent 必須**推理拿戒指的規則**：
   - 從前世記憶看出「作為 1, 2, 3, 4, 5 號時拿戒指都失敗並死亡」
   - 或觀察當前輪迴中其他 pov 拿戒指的結果
   - 推論：「只有 6 號能拿」
5. Agent 必須**時機拿捏**：
   - 6 燈齊亮後搶先觸碰戒指
   - 太早 → 其他 pov 還沒按對按鈕（6 燈未齊，防護窗未開）
   - 太晚 → 其他 pov（按劇本）可能搶先觸碰造成錯亂 → pov_6 雖活但關卡失敗

## 系統不變量（Invariants）

為保證「不會形成邏輯錯亂」（使用者第 3 點原始需求），系統必須維持以下不變量：

- **INV-1 唯一事實來源**：World Engine 是世界狀態的唯一持有者；所有 pov 透過 Observation/Action JSON 介面互動，不得直接改寫 world state
- **INV-2 Immutable 劇本**：`script_n` 一旦生成完畢並通過驗證即不可修改；執行時 pov_n（n<6）嚴格按腳本行動
- **INV-3 因果閉環**：`script_n` 與 `script_{n-1}` 的共有事件必須完全一致（劇本生成時自動驗證；不通過即 retry 或人類介入）
- **INV-4 pov_6 自由**：pov_6 不受腳本約束，但其行為不影響任何 pov 的前世記憶（因為無 pov_7）
- **INV-5 觀察對稱性**：每個 pov 看不見自己的號碼牌、規則、目標；可看見他人號碼牌、物理環境、公開對話
- **INV-6 事件日誌可重放**：關卡執行時所有事件 append-only 寫入 event log；給定初始狀態 + 所有劇本 + event log，可完整重建任意時刻的 world state
- **INV-7 單一自由主體**：任一 tick 有且僅有一個 pov 提交「自由意志」動作（即 pov_6）；其他 pov 的動作來自劇本
- **INV-8 無時間悖論**：關卡執行中不得出現「違反劇本」的行為鏈（pov_1~5 按腳本走是硬約束）

## 待 Coordinator 於 Propose 階段展開的議題

以下未決議，應由 Coordinator 在 `/opsx:propose` 產出 spec 時細化（或記入 `open questions`）：

1. **關卡物理空間**
   - 2D 格子（簡單）vs. 2D 連續座標 vs. 3D？
   - 房間尺寸、按鈕區域、戒指位置、6 body 起始位置
2. **互動原語清單**
   - `observe`, `speak(msg)`, `move(dir)`, `press(button_id)`, `touch_ring`, `wait` 的具體參數、前置條件與結果
3. **按鈕與戒指機制細節**
   - 按對 vs 按錯的判定（是否有 delay？）
   - 防護窗開啟動畫（瞬時 vs 漸進？）
   - 搶戒指的競爭解算（同時觸碰時誰得到？）
4. **劇本生成的具體實作**
   - 使用哪個 LLM 生成 `script_1~5`（Sonnet 4.7？Haiku 4.5？）
   - 人格配置檔案格式（YAML？）
   - 生成成本與時間預估
   - 驗證不通過時的重試策略與上限 K 值
5. **關卡執行期的即時對話**
   - pov_1~5 遇到 pov_6 主動對話時如何即時回應（限制在人格範圍）
   - 即時對話是否寫回 event log（與劇本疊加）
6. **pov_6 的 prompt 結構**
   - System prompt：persona + 規則約束 + 前世記憶（應套用 prompt caching）
   - User prompt：每回合 observation
   - Caching 邊界設計
7. **失敗後的 retry 策略**
   - pov_6 拿錯戒指 or timeout 後是否允許重玩？
   - 重玩時 pov_6 是否保留本次記憶？
8. **成本與效能預估**
   - 劇本生成階段（5 次 LLM call + 驗證 retry）
   - 關卡執行階段（pov_6 每回合 1 次 + pov_1~5 可能的即時對話）
   - Prompt caching 能省多少
9. **測試策略**
   - 單元測試：劇本閉環驗證、按鈕規則、戒指觸碰判定
   - 整合測試：完整關卡跑一遍（含 LLM stub）
   - `FakeAnthropicClient` 作為單元測試替身
10. **全域規範的落實**
    - `docker/`（Dockerfile、build.sh、docker-compose.yaml）
    - `logs/`（.gitkeep，並 gitignore `logs/*`）
